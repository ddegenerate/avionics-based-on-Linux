## 第一步：构建航电靶标程序

**目标:** 编写一个对时间敏感的单线程 C 程序，模拟航电系统的飞控逻辑，并能测量执行时间。

**操作步骤:**
编写 `avionics_target.c`，使用高精度时钟 `CLOCK_MONOTONIC` 计算 10000 次 50x50 矩阵乘法的执行时间。

### `avionics_target.c`

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <time.h>
#include <stdint.h>

#define NUM_ITERATIONS 10000
#define MATRIX_SIZE 50
//矩阵乘法
void simulate_flight_control_logic(float A[MATRIX_SIZE][MATRIX_SIZE], 
                                   float B[MATRIX_SIZE][MATRIX_SIZE], 
                                   float C[MATRIX_SIZE][MATRIX_SIZE]) 
{
    for (int i = 0; i < MATRIX_SIZE; i++) 
    {
        for (int j = 0; j < MATRIX_SIZE; j++) 
        {
            C[i][j] = 0.0;
            for (int k = 0; k < MATRIX_SIZE; k++) 
            {
                C[i][j] += A[i][k] * B[k][j];
            }
        }
    }
}

int main() {
    struct timespec start, end;  //结构体里面有s和ns相当于就是计现在是多少秒零多少纳秒
    uint64_t elapsed_us;
    uint64_t max_time_us = 0;
    uint64_t total_time_us = 0;

    // 初始化矩阵
    float A[MATRIX_SIZE][MATRIX_SIZE];
    float B[MATRIX_SIZE][MATRIX_SIZE];
    float C[MATRIX_SIZE][MATRIX_SIZE];

    // 填充随机数据
    for (int i = 0; i < MATRIX_SIZE; i++) 
    {
        for (int j = 0; j < MATRIX_SIZE; j++) 
        {
            A[i][j] = (float)(rand() % 100) / 10.0;
            B[i][j] = (float)(rand() % 100) / 10.0;
        }
    }
    printf("=> 矩阵大小: %dx%d, 循环次数: %d\n", MATRIX_SIZE, MATRIX_SIZE, NUM_ITERATIONS);
    // 先空跑10次达到稳态
    for (int i = 0; i < 10; i++) 
    {
        simulate_flight_control_logic(A, B, C);
    }

    // 正式测试循环
    for (int iter = 0; iter < NUM_ITERATIONS; iter++) {
        // 1. 获取循环开始时间
        clock_gettime(CLOCK_MONOTONIC, &start);  //CLOCK_MONOTONIC是单调递增时钟，修改系统时间不影响
        // 2. 执行飞行控制逻辑
        simulate_flight_control_logic(A, B, C);
        // 3. 获取循环结束时间
        clock_gettime(CLOCK_MONOTONIC, &end);
        // 4. 计算耗费的微秒 (us)
  
        elapsed_us = (end.tv_sec - start.tv_sec) * 1000000 + 
                     (end.tv_nsec - start.tv_nsec) / 1000;

        // 5. 记录所有次数里面的最长运行时间，算是伪WCET
        if (elapsed_us > max_time_us) 
        {
            max_time_us = elapsed_us;
        }
        // 累加时间用于计算平均值
        total_time_us += elapsed_us;
    }

    // 计算平均执行时间
    uint64_t avg_time_us = total_time_us / NUM_ITERATIONS;
    // 打印测试结果
    printf("\n测试完成！\n");
    printf("------------------------------------------------\n");
    printf("平均执行时间 (Average): %llu 微秒 (us)\n", (unsigned long long)avg_time_us);
    printf("最坏情况执行时间 (WCET):  %llu 微秒 (us)\n", (unsigned long long)max_time_us);
    printf("系统抖动 (Jitter):      %llu 微秒 (us)\n", (unsigned long long)(max_time_us - avg_time_us));
    printf("------------------------------------------------\n");
    // 防优化机制：打印矩阵中的一个值，防止聪明的现代编译器把没有输出的矩阵运算直接删掉 (Dead Code Elimination)
    printf("[防优化输出] C[0][0] = %f\n", C[0][0]);
    return 0;
}
```

代码里的最坏执行时间并不是严格意义上的经过演算方法得到的严格最坏时间，而是10000次运行次数里面时间最长的一个伪最坏执行时间。

编译命令：

```bash
gcc -O0 avionics_target.c -o avionics_sensor
```

`-O0` 表示不进行任何优化，防止把矩阵空乘法直接优化掉。

## 第三步：核心隔离

**目标:** 弄清楚CPU物理核和逻辑核，将航电任务绑定至独立物理核，并通过内核参数将其从通用调度域中彻底剔除。

### 3.1 识别 CPU 物理核和逻辑核

6核12线程 = 6个物理核，但操作系统看到12个逻辑核。使用 `lscpu` 探明系统拥有 6 个物理核。为了防止超线程（SMT）导致的微架构资源争抢，必须将一整个物理核心（即两个逻辑核）完全隔离。

查询逻辑核 11 的物理兄弟核：
```bash
cat /sys/devices/system/cpu/cpu11/topology/thread_siblings_list
# 结果: 10-11
```
**结论:** 逻辑核 10 和 11 共享同一个物理核，现将其共同划分为一个专用分区。

### 3.2 核心隔离分区
修改 GRUB 配置文件，在内核启动参数中加入 `isolcpus`，从内核调度器中剥离该物理核。
```bash
sudo nano /etc/default/grub
# 修改为: GRUB_CMDLINE_LINUX_DEFAULT="quiet splash isolcpus=10,11"
# 注：这里根据前面查询的兄弟核来填，兄弟核不一定是相邻的
# 结束后记得把这里的isolcpus=10,11删掉，取消隔离配置
sudo update-grub
sudo reboot
```

### 3.3 核心隔离后压力测试
验证隔离区生效 (`cat /sys/devices/system/cpu/isolated` 成功显示 `10-11`)。
* **终端 1 (公共区高负载):** `stress-ng --cpu 10 --cache 2 --vm 2 --timeout 60s`

* **终端 2 (隔离分区运行程序):** `taskset -c 11 ./avionics_sensor`，`taskset -c 10 ./avionics_sensor`，`taskset -c 9 ./avionics_sensor`

## 第四步：利用SCHED_DEADLINE调度类模拟ARINC 653时间分区

**目标:** 在物理隔离的基础上，为复现航电 ARINC 653 标准中的时间窗口机制，使用SCHED_DEADLINE调度类。

**思路:** 通过 `SCHED_DEADLINE` 调度类为任务配置 **运行时间 (Runtime)**、**截止期限 (Deadline)** 和 **周期 (Period)**。Linux 底层的 CBS（恒定带宽服务器）算法会严格保障这一窗口，并在预算耗尽时强制挂起任务，从而防止死循环拖死整机。

### 4.1 编写653时隙调度程序 (`avionics_sche.c`)

标准 C 库 (`glibc`) 没有直接封装 `SCHED_DEADLINE`，我们需要直接调用底层的系统调用 `sched_setattr`。

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <time.h>
#include <stdint.h>
#include <unistd.h>
#include <sys/syscall.h>
#include <linux/sched.h>

#define NUM_ITERATIONS 10000
#define MATRIX_SIZE 50

// 定义 SCHED_DEADLINE 需要的底层数据结构
struct sched_attr {
    uint32_t size;
    uint32_t sched_policy;
    uint64_t sched_flags;
    int32_t sched_nice;
    uint32_t sched_priority;
    uint64_t sched_runtime; //运行时
    uint64_t sched_deadline;//周期时
    uint64_t sched_period;  //截止时
};

// 封装底层的系统调用，c语言没有这个函数，去内核调用
// 这个函数就是把哪个进程改成什么调度策略
int sched_setattr(pid_t pid, const struct sched_attr *attr, unsigned int flags) {
    return syscall(SYS_sched_setattr, pid, attr, flags);
}

// 模拟飞行控制逻辑（矩阵乘法保持不变）
void simulate_flight_control_logic(float A[MATRIX_SIZE][MATRIX_SIZE], 
                                   float B[MATRIX_SIZE][MATRIX_SIZE], 
                                   float C[MATRIX_SIZE][MATRIX_SIZE]) {
    for (int i = 0; i < MATRIX_SIZE; i++) {
        for (int j = 0; j < MATRIX_SIZE; j++) {
            C[i][j] = 0.0;
            for (int k = 0; k < MATRIX_SIZE; k++) {
                C[i][j] += A[i][k] * B[k][j];
            }
        }
    }
}

int main() {
    // ARINC 653的时间窗口 (SCHED_DEADLINE)
    struct sched_attr attr;
    attr.size = sizeof(attr);
    attr.sched_flags = 0;
    attr.sched_nice = 0;
    attr.sched_priority = 0;
    
    // 设置调度策略为 SCHED_DEADLINE
    // 宏定义 6 对应 SCHED_DEADLINE
    attr.sched_policy = 6; 

    // 参数计算 (基于你之前测试的 WCET 数据)：
    // 之前隔离状态下的 WCET 大约是 2 毫秒 (2000 us = 2,000,000 ns)
    // 我们预留一点余量，申请 3 毫秒的运行时间。
    
    // 1 毫秒 (ms) = 1,000,000 纳秒 (ns)
    attr.sched_runtime  = 3 * 1000000;   // 运行时间：3 毫秒
    attr.sched_deadline = 10 * 1000000;  // 截止期限：10 毫秒
    attr.sched_period   = 10 * 1000000;  // 周期：10 毫秒

    // 应用调度策略 (0 代表当前进程)
    if (sched_setattr(0, &attr, 0) == -1) {
        perror("设置 SCHED_DEADLINE 失败 (必须使用 sudo 运行)");
        exit(EXIT_FAILURE);
    }
    printf("=> 已成功切换至 SCHED_DEADLINE (时间分区) 调度策略\n");
    printf("=> 时间合同: 每 %llu ms 运行 %llu ms\n", 
            (unsigned long long)(attr.sched_period / 1000000), 
            (unsigned long long)(attr.sched_runtime / 1000000));
    // ==========================================================

    struct timespec start, end;
    uint64_t elapsed_us, max_time_us = 0, total_time_us = 0;
    float A[MATRIX_SIZE][MATRIX_SIZE], B[MATRIX_SIZE][MATRIX_SIZE], C[MATRIX_SIZE][MATRIX_SIZE];

    for (int i = 0; i < MATRIX_SIZE; i++) {
        for (int j = 0; j < MATRIX_SIZE; j++) {
            A[i][j] = (float)(rand() % 100) / 10.0;
            B[i][j] = (float)(rand() % 100) / 10.0;
        }
    }

    printf("=> 开始运行航电靶标程序\n");
    for (int i = 0; i < 10; i++) simulate_flight_control_logic(A, B, C);

    for (int iter = 0; iter < NUM_ITERATIONS; iter++) {
        clock_gettime(CLOCK_MONOTONIC, &start);
        simulate_flight_control_logic(A, B, C);
        clock_gettime(CLOCK_MONOTONIC, &end);

        elapsed_us = (end.tv_sec - start.tv_sec) * 1000000 + (end.tv_nsec - start.tv_nsec) / 1000;
        if (elapsed_us > max_time_us) max_time_us = elapsed_us;
        total_time_us += elapsed_us;
    }

    printf("\n------------------------------------------------\n");
    printf("平均执行时间 (Average): %llu 微秒 (us)\n", (unsigned long long)(total_time_us / NUM_ITERATIONS));
    printf("最坏情况执行时间 (WCET):  %llu 微秒 (us)\n", (unsigned long long)max_time_us);
    printf("系统抖动 (Jitter):      %llu 微秒 (us)\n", (unsigned long long)(max_time_us - (total_time_us / NUM_ITERATIONS)));
    printf("------------------------------------------------\n");
    printf("[防优化输出] C[0][0] = %f\n", C[0][0]);

    return 0;
}
```

编译命令：

```bash
gcc -O0 avionics_sche.c -o avionics_sche
```

## 第五步：出现内核调度器冲突

### 5.1 `taskset` 绑定失败
尝试像之前一样，使用 `taskset` 将这个 `SCHED_DEADLINE` 程序绑定到被 `isolcpus` 隔离的核心 11 上运行时，遭遇了内核级的直接拒绝：

```bash
sudo taskset -c 11 ./avionics_sche
# 报错: 设置 SCHED_DEADLINE 失败 (必须使用 sudo 运行): Operation not permitted
```

**为什么？**
1. **准入控制器 ：** `SCHED_DEADLINE` 必须在全局调度域中计算总带宽以防违约。使用 `taskset` 强行限制核心，会导致准入控制器无法完成算力担保，从而触发自我保护拒绝提权。
2. **`isolcpus` 破坏了根调度域：** 在 GRUB 中配置 `isolcpus=10,11`，实际上是将这两个核心从内核的"根调度域"中开除了，内核无法找到。

## 第六步：寻找解决方法

**目标:** 解决 `taskset` 无法绑定 `SCHED_DEADLINE` 的问题。

**方案:** 问题本质上是 `isolcpus` 与 `SCHED_DEADLINE` 的冲突，所以这两个东西得选一个舍弃掉。这里弃用 `isolcpus`，改用 `cgroups v2`。

### 6.1 撤销 `isolcpus` 隔离
将核心 10 和 11 的控制权交还给系统调度器（保留无 Tick 模式）。
```bash
sudo nano /etc/default/grub
# 修改为: GRUB_CMDLINE_LINUX_DEFAULT="quiet splash nohz_full=10,11 rcu_nocbs=10,11"
sudo update-grub
sudo reboot
```

### 6.2 构建分区调度域
重启后进入 `/sys/fs/cgroup` 目录，注意到有以下内容：

![隔离核心与非隔离核心结果](https://free.boltp.com/2026/05/19/6a0bee1c47be2.webp)

主要列一下比较关键的几个东西：

**cgroups 要改的几个文件**
- `cgroup.subtree_control`：父级控制子级可用哪些控制器（`+cpuset` 表示授权子目录使用 CPU 绑定能力）
- `cpuset.cpus`：限制当前 cgroup 内的进程只能使用指定的 CPU 核心
- `cpuset.cpus.partition`：将指定核心从系统全局调度器中切出，形成独立调度域`root`状态意味着拥有专属运行队列，不再参与全局负载均衡）

可以通过cat指令看一下各个文件的内容
![隔离核心与非隔离核心结果](https://free.boltp.com/2026/05/19/6a0bf0338126e.webp) 

**为什么要这样做？**  
 `cpuset.cpus.partition = root` 会将核心 10/11 从全局调度器中物理隔离，使该分区内的进程拥有独立的运行队列和算力账本，从而允许设置 `SCHED_FIFO` 等实时策略，避免 `operation not permitted`。
 
**1. 激活 cpuset 资源控制器：**
```bash
cd /sys/fs/cgroup
sudo bash -c 'echo "+cpuset" > cgroup.subtree_control'
```
cgroup.subtree_control文件中出现有cpuset即为成功，如下图所示
![隔离核心与非隔离核心结果](https://free.boltp.com/2026/05/19/6a0bf4692a83c.webp)

**2. 创建分区：**
```bash
sudo mkdir avionics_partition
cd avionics_partition
```
说明：在cgroups v2 中，mkdir 创建子目录即自动生成一个子 cgroup。内核会在该目录内自动填充必要的接口文件（如 cgroup.procs、cgroup.controllers、cpuset.cpus 等），无需手动创建任何文件。如下图所示
![隔离核心与非隔离核心结果](https://free.boltp.com/2026/05/19/6a0bf2b234f0a.webp) 


**3. 给分区划拨物理核心：** 规定该分区独占 CPU 10 和 11。
```bash
sudo bash -c 'echo "10-11" > cpuset.cpus'
```
和上面一样，在敲完指令后可以cat一下对应文件，看看内容有没有加进去

**4. 把该cgroup设置为隔离的根调度域 ：**
```bash
sudo bash -c 'echo "root" > cpuset.cpus.partition'
```
*注：此操作切断了核心 10/11 与全局的负载均衡，状态由 `member` 跃迁为 `root`。*

**5. 把当前终端直接加入分区 ：**
在终端上运行的任务自动在分区中运行
```bash
#获取当前终端的PID
echo $$
#将PID写入cgroup的cgroups.procs文件
sudo sh -c "echo 你的PID > /sys/fs/cgroup/avionics_partition/cgroup.procs"
#通过查看当前shell的cgroup路径
cat /proc/self/cgroup
#如果输出0::/avionics_partition的话就说明终端已经放进该分区里了
cat /proc/self/status | grep Cpus_allowed_list
#查看核心是否隔离，我这里隔离10-11,所以应该输出Cpus_allowed_list: 10-11
```

## 第七步：可视化验证

**目标:** 使用 `trace-cmd` 录制内核调度事件（`sched_switch` 和 `sched_wakeup`），并通过 KernelShark 图形化工具直观验证核心隔离、`SCHED_DEADLINE` 的 3ms/7ms 时间窗口，以及整体的核心隔离效果。

### 7.1 安装追踪工具

首先，在系统中安装所需的追踪与可视化分析工具：

```bash
sudo apt update
sudo apt install trace-cmd kernelshark
```

### 7.2 新建终端准备录制

为避免 `trace-cmd` 的录制操作本身产生调度事件从而污染隔离区的数据，这里需要新建一个普通终端。输入以下命令开始录制底层调度事件：

```bash
sudo trace-cmd record \
    -e sched:sched_switch \
    -e sched:sched_wakeup \
    -e sched:sched_wakeup_new
```

**参数解释：**
此命令使用 `trace-cmd` 工具记录 Linux 内核调度器相关的三个核心 tracepoint 事件，并将结果保存到当前目录下的 `trace.dat` 文件中。

1. **`sched_switch` (上下文切换)：** 记录了任务真正拿到 CPU（上台）和被剥夺 CPU（下台）的绝对时间点。两次 switch 之间的时间差，就是程序占据核心的真实运行时间。
2. **`sched_wakeup` (唤醒)：** 记录进程从休眠状态（如等待内存的 D 状态，或强制休眠的 S 状态）被唤醒，并放入就绪队列的瞬间。结合 switch 事件，可以精确算出内核的调度延迟。
3. **`sched_wakeup_new` (新进程诞生)：** 由于在终端敲击回车时会 fork 出全新的进程，如果不加此参数，追踪工具就会错过程序第一秒的启动瞬间。

### 7.3 启动程序与结束录制

* **终端 A (监控端)：** 新建的正在运行 `trace-cmd` 录制命令的新终端。
* **终端 B (执行端)：** 已经成功被关进 `cgroups` 隔离区 (CPU 10-11) 的操作终端。

**执行步骤：**

1. 确认 **终端 A** 已处于持续录制状态。
2. 切换到 **终端 B**，运行对应的测试程序：

```bash

# 或者测试 ARINC 653 时间分区状态

```

3. 等待 **终端 B** 中的程序执行完毕并打印出结果。

## 第八步：内存分区（CMA 连续物理内存隔离）

**目标:** 为航电分区分配一块专属的连续大块物理内存。

**为什么弃用 `memmap`：** 原方案打算通过 GRUB 的 `memmap=512M$0x110000000` 在 4GB 以上物理地址挖一块出来，但在 Ryzen 5500U APU 上实测连系统都起不来。原因应该是现代 UEFI 固件对 e820 内存映射有严格校验，加上 AMD IOMMU 的保护，启动阶段硬改内存布局直接触发 kernel panic。

**换思路:** 改用 Linux 内核的 **CMA（Contiguous Memory Allocator，连续内存分配器）**。CMA 只在启动时做个"软标记"——把一段物理页标为可迁移区域，平时伙伴系统可以临时借给可移动页面用（比如页缓存），等 DMA API 真正来申请时，内核自动把占用页面挪走，还你一块连续物理内存。这样既不跟 e820 硬碰硬，隔离效果又跟 `memmap` 一样。

---

### 8.1 开启 CMA 预留（GRUB 配置）

在 GRUB 中追加 `cma=预留大小`：

```bash
sudo nano /etc/default/grub
# 在原有参数末尾追加 cma=520M，例如：
# GRUB_CMDLINE_LINUX_DEFAULT="quiet splash nohz_full=10,11 rcu_nocbs=10,11 cma=520M"
sudo update-grub
sudo reboot
```

> **一个小坑：** 把 CMA 总池子设为 512M 时，实际上永远申请不到完整的 512M。因为池子里面有一部分会被内核元数据占用，真正能拿出来用的比 512M 少一点。我一开始设 512M 就直接报错了，后来改成 520M 才正常。所以建议留一点余量。

重启后验证：

```bash
sudo dmesg | grep -i cma
```

![CMA 预留结果](https://img.cdn1.vip/i/6a1559f4d3ea7_1779784180.webp)

从输出看，系统在 4GB 边界之上（0x100000000）预留了 520MB 的 CMA 区域。驱动实际分配时，DMA 引擎跳过了开头 2MB 的对齐保留区，最终可用内存从 0x100200000 开始。

> **说明：** CMA 的具体物理地址由内核启动时自己选，用户没法指定也无需指定。反正航电场景只需要"一块连续隔离的内存"，不在乎它具体落在哪个物理地址上。

---

### 8.2 编写 CMA 驱动

驱动的作用是从 CMA 池子里申请 512MB 连续物理内存，然后创建 `/dev/avionics_cma` 设备节点，用户态程序通过它 `mmap` 过来用。

```bash
mkdir -p cma_avionics && cd cma_avionics
```

**`avionics_cma.c`：**

```c
#include <linux/module.h>
#include <linux/platform_device.h>
#include <linux/dma-mapping.h>
#include <linux/miscdevice.h>
#include <linux/fs.h>
#include <linux/slab.h>

#define CMA_MEM_SIZE (512 * 1024 * 1024) // 512MB

static void *cma_cpu_addr = NULL;
static dma_addr_t cma_dma_handle;
static struct platform_device *cma_pdev;

// DMA 掩码声明为静态全局变量，防止 init 返回后栈指针悬空
static u64 cma_dma_mask = DMA_BIT_MASK(64);

// 处理用户态的 mmap 请求
static int cma_mmap(struct file *filp, struct vm_area_struct *vma)
{
    unsigned long size = vma->vm_end - vma->vm_start;

    // 严格的边界与偏移校验，防止用户态恶意越界触发内核 Oops
    if (vma->vm_pgoff != 0) {
        printk(KERN_WARNING "Avionics CMA: 拒绝非零偏移的 mmap 请求\n");
        return -EINVAL;
    }
    if (size > CMA_MEM_SIZE) {
        printk(KERN_WARNING "Avionics CMA: 映射请求超限 (%lu > %lu)\n",
               size, (unsigned long)CMA_MEM_SIZE);
        return -EINVAL;
    }

    // 使用 DMA API 标准映射函数，自动处理缓存一致性与物理连续性
    return dma_mmap_coherent(&cma_pdev->dev, vma, cma_cpu_addr,
                             cma_dma_handle, size);
}

static const struct file_operations cma_fops = {
    .owner = THIS_MODULE,
    .mmap  = cma_mmap,
};

static struct miscdevice cma_misc = {
    .minor = MISC_DYNAMIC_MINOR,
    .name  = "avionics_cma",   // 设备节点: /dev/avionics_cma
    .fops  = &cma_fops,
};

static int __init avionics_cma_init(void)
{
    int ret;

    // 1. 分配并注册虚拟平台设备
    cma_pdev = platform_device_alloc("avionics_cma_dev", PLATFORM_DEVID_NONE);
    if (!cma_pdev) {
        printk(KERN_ERR "Avionics CMA: 分配 platform_device 失败\n");
        return -ENOMEM;
    }

    // 2. 绑定 DMA 掩码并添加进内核设备模型
    cma_pdev->dev.dma_mask = &cma_dma_mask;
    cma_pdev->dev.coherent_dma_mask = cma_dma_mask;

    ret = platform_device_add(cma_pdev);
    if (ret) {
        printk(KERN_ERR "Avionics CMA: 注册 platform_device 失败\n");
        platform_device_put(cma_pdev);
        return ret;
    }

    // 3. 从 CMA 区域分配连续物理内存
    cma_cpu_addr = dma_alloc_coherent(&cma_pdev->dev, CMA_MEM_SIZE,
                                      &cma_dma_handle, GFP_KERNEL);
    if (!cma_cpu_addr) {
        printk(KERN_ERR "Avionics CMA: 无法分配 %lu MB 的连续内存 "
               "(请检查 GRUB 中是否已添加 cma=520M)\n",
               (unsigned long)CMA_MEM_SIZE / (1024 * 1024));
        platform_device_unregister(cma_pdev);
        return -ENOMEM;
    }

    // 4. 注册字符设备供用户态访问
    ret = misc_register(&cma_misc);
    if (ret) {
        printk(KERN_ERR "Avionics CMA: 字符设备注册失败\n");
        dma_free_coherent(&cma_pdev->dev, CMA_MEM_SIZE,
                          cma_cpu_addr, cma_dma_handle);
        platform_device_unregister(cma_pdev);
        return ret;
    }

    printk(KERN_INFO "Avionics CMA: 成功分配 512MB 连续物理内存, "
           "物理地址: %pad\n", &cma_dma_handle);
    return 0;
}

static void __exit avionics_cma_exit(void)
{
    misc_deregister(&cma_misc);
    if (cma_cpu_addr) {
        dma_free_coherent(&cma_pdev->dev, CMA_MEM_SIZE,
                          cma_cpu_addr, cma_dma_handle);
    }
    platform_device_unregister(cma_pdev);
    printk(KERN_INFO "Avionics CMA: 模块已卸载，内存与设备已清理\n");
}

module_init(avionics_cma_init);
module_exit(avionics_cma_exit);
MODULE_LICENSE("GPL");
```

### 8.3 用户态接管 CMA 内存

修改靶标程序，把原来的栈/堆分配换成从 `/dev/avionics_cma` 映射 CMA 内存。

**`avionics_mem.c`：**

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <fcntl.h>
#include <sys/mman.h>
#include <unistd.h>
#include <string.h>
#include <stdint.h>
#include <time.h>
#include <sys/mman.h>

#define NUM_ITERATIONS 10000
#define MATRIX_SIZE 50
#define ALLOC_SIZE  (2 * 1024 * 1024)   // 每块矩阵区域 2MB
#define TOTAL_SIZE  (512 * 1024 * 1024) // 必须与驱动申请的 CMA 大小一致

// 模拟飞行控制逻辑（矩阵乘法）
void simulate_flight_control_logic(float A[MATRIX_SIZE][MATRIX_SIZE],
                                   float B[MATRIX_SIZE][MATRIX_SIZE],
                                   float C[MATRIX_SIZE][MATRIX_SIZE])
{
    for (int i = 0; i < MATRIX_SIZE; i++) {
        for (int j = 0; j < MATRIX_SIZE; j++) {
            C[i][j] = 0.0;
            for (int k = 0; k < MATRIX_SIZE; k++) {
                C[i][j] += A[i][k] * B[k][j];
            }
        }
    }
}

// 从内核 CMA 驱动映射物理连续内存
void* map_cma_memory(void)
{
    int fd = open("/dev/avionics_cma", O_RDWR);
    if (fd < 0) {
        perror("无法打开 /dev/avionics_cma");
        exit(EXIT_FAILURE);
    }

    void *ptr = mmap(NULL, TOTAL_SIZE, PROT_READ | PROT_WRITE,
                     MAP_SHARED, fd, 0);
    close(fd);

    if (ptr == MAP_FAILED) {
        perror("CMA mmap 失败");
        exit(EXIT_FAILURE);
    }
    return ptr;
}

int main()
{
    // 锁定全部虚拟地址空间，防止被换出到 Swap
    if (mlockall(MCL_CURRENT | MCL_FUTURE) == -1) {
        perror("内存锁定失败");
        exit(EXIT_FAILURE);
    }

    void *base = map_cma_memory();

    // 预热：强制触碰前 6MB，让 MMU 提前建好页表，消除首次访问的缺页开销
    memset(base, 0, 3 * ALLOC_SIZE);

    // 在连续物理内存上切分出 A、B、C 三块
    float (*A)[MATRIX_SIZE] = (float (*)[MATRIX_SIZE])(base + 0 * ALLOC_SIZE);
    float (*B)[MATRIX_SIZE] = (float (*)[MATRIX_SIZE])(base + 1 * ALLOC_SIZE);
    float (*C)[MATRIX_SIZE] = (float (*)[MATRIX_SIZE])(base + 2 * ALLOC_SIZE);

    printf("=> 航电系统：已成功接管 512MB 专属物理连续内存 (基于 CMA)\n");

    // 填充随机数据
    for (int i = 0; i < MATRIX_SIZE; i++) {
        for (int j = 0; j < MATRIX_SIZE; j++) {
            A[i][j] = (float)(rand() % 100) / 10.0;
            B[i][j] = (float)(rand() % 100) / 10.0;
        }
    }

    // ---- 时间测量与主循环 ----
    struct timespec start, end;
    uint64_t elapsed_us, max_time_us = 0, total_time_us = 0;

    printf("=> 矩阵大小: %dx%d, 循环次数: %d\n", MATRIX_SIZE, MATRIX_SIZE, NUM_ITERATIONS);

    // 空跑 10 次达到稳态
    for (int i = 0; i < 10; i++)
        simulate_flight_control_logic(A, B, C);

    // 正式测试循环
    for (int iter = 0; iter < NUM_ITERATIONS; iter++) {
        clock_gettime(CLOCK_MONOTONIC, &start);
        simulate_flight_control_logic(A, B, C);
        clock_gettime(CLOCK_MONOTONIC, &end);

        elapsed_us = (end.tv_sec - start.tv_sec) * 1000000 +
                     (end.tv_nsec - start.tv_nsec) / 1000;
        if (elapsed_us > max_time_us)
            max_time_us = elapsed_us;
        total_time_us += elapsed_us;
    }

    uint64_t avg_time_us = total_time_us / NUM_ITERATIONS;
    printf("\n测试完成！\n");
    printf("------------------------------------------------\n");
    printf("平均执行时间 (Average): %llu 微秒 (us)\n", (unsigned long long)avg_time_us);
    printf("最坏情况执行时间 (WCET):  %llu 微秒 (us)\n", (unsigned long long)max_time_us);
    printf("系统抖动 (Jitter):      %llu 微秒 (us)\n",
           (unsigned long long)(max_time_us - avg_time_us));
    printf("------------------------------------------------\n");
    printf("[防优化输出] C[0][0] = %f\n", C[0][0]);

    return 0;
}
```

### 8.4 架构边界说明

这套方案实现了宏观的物理内存隔离，但也不是万能的：

| 能力 | 状态 | 说明 |
| :--- | :--- | :--- |
| 防止其他进程占用这块内存 | 成功 | `dma_alloc_coherent` 后页面从伙伴系统移出，不可迁移 |
| 保证数据落在连续物理地址上 | 成功 | CMA + DMA API 天然保证物理连续性 |
| x86 平台内存访问性能 | 无退化 | x86 是缓存一致架构，`dma_alloc_coherent` 返回的是普通 cached 内存 |
| 防止航电进程自身野指针越界 | 不支持 | 这个需要 MMU/MPU 硬件级保护，Linux 用户态做不了 |
| 控制精确的物理起始地址 | 不支持 | CMA 地址由内核自动选，没法指定 |

## 第九步：CMA 内存分区效果验证

**目标:** 验证 CMA 内存确实被隔离了，并且在系统高内存压力下程序表现稳定。

**思路:** 内存分区不像 CPU 调度那样有 `sched_switch` 可追踪，所以验证从三个角度来：静态确认 CMA 使用状况、缺页事件对比、压力下的抖动对比。

---

### 9.1 静态验证：查看 CMA 使用状态


![CMA debugfs 验证](https://img.cdn1.vip/i/6a1560ad23a67_1779785901.webp)

在现代内核（Ubuntu 24.04 的 6.x）中，GRUB 的 `cma=520M` 预留的区域默认挂载为 `reserved` 目录，说明 CMA 在启动时已正常工作。

换算一下：`133120 页 * 4KB = 532480 KB = 520 MB`，跟 GRUB 里设的 520M 完全对得上。

已分配页数：`131510 页 * 4KB = 526040 KB ≈ 513.7 MB`。比 520MB 少了约 6MB，这是内核管理该区域元数据以及 DMA 2MB 起始对齐的正常开销。

### 9.3 抗干扰验证：内存踩踏测试

**结果：**

无负载下 CMA 版本：

| **CMA 内存（无压力）** | 455 us | 146 us |

## 第十步：L3 缓存分区

**目标:** 前面用 cgroups 解决了"谁能跑在核心 10/11 上"的调度问题，CMA 解决了"数据放在哪块物理内存"的问题。还有缓存分区的问题没有解决。这一步通过 Linux 的 `resctrl` 接口调用 Intel/AMD 的 CAT（Cache Allocation Technology）硬件功能，为核心 10/11 划出专属的 L3 缓存块。

> **说明:** cgroups (cpuset) 管的是软件调度层面，`resctrl` 管的是硬件微架构层面。

### 10.1 挂载 resctrl 并确认硬件能力

先把硬件资源分配接口挂起来，看看这颗 U 支不支持 CAT：

```bash
sudo mount -t resctrl resctrl /sys/fs/resctrl
cd /sys/fs/resctrl
ls info/
```

![info目录内容](https://img.cdn1.vip/i/6a165b9eda74d_1779850142.png)

可以看到，我这里的`info/` 目录下有四个条目：

| 条目 | 用途 |
| :--- | :--- |
| **L3** | L3 缓存分配（CAT），做缓存物理分区的核心接口 |
| **L3_MON** | L3 缓存监控（RDT），可实时看进程占了多少缓存 |
| **MB** | 内存带宽分配（MBA），可限制非关键任务的访存带宽 |
| **last_cmd_status** | 排错文件，每次写入操作后刷新，显示 ok 或具体失败原因 |

查看 L3 缓存的容量位掩码（CBM）：

```bash
cat info/L3/cbm_mask
```

这里我的结果是`ffff` = 16 个 bit。5500U 有 8MB L3 缓存，每个 bit 控制 `8MB / 16 = 512KB`。计划把最右侧 4 个 bit（`f`，共 2MB）划出来给航电专属。

### 10.2 修改全局掩码（避坑）

一个容易踩的坑：默认情况下根目录的 `schemata` 对 L3 是全量开放的（`ffff`），如果直接创建新组并分配，内核会报 `Schemata overlaps`，拒绝开启独占模式。

此外这颗 U 有两个 L3 逻辑节点（L3:0 和 L3:1），两边的掩码都得退让：

```bash
sudo bash -c 'echo "L3:0=fff0;1=fff0" > /sys/fs/resctrl/schemata'
```

这一步把系统全局进程限制在前 12 个 bit（`fff0`），把最后 4 个 bit 彻底腾出来。

> **排错经历：** 我最开始只改了 L3:0=fff0，漏了 L3:1 还是 ffff。写 `exclusive` 时报错 `无效的参数`，查 `cat info/last_cmd_status` 显示 `Schemata overlaps`，定位到是节点 1 还占着那 4 个 bit。两边都改成 fff0 后就正常了。

![缓存分区报错](https://img.cdn1.vip/i/6a165c3521b16_1779850293.webp)

![缓存分区报错解决](https://img.cdn1.vip/i/6a165d2ecb2ae_1779850542.webp)

### 10.3 创建航电专属资源组并设为独占

```bash
# 1. 创建资源组
sudo mkdir /sys/fs/resctrl/avionics_cache

# 2. 分配专属掩码（分区暂时挂在 L3:0 上，L3:1 不做分配）
sudo bash -c 'echo "L3:0=f;1=0" > /sys/fs/resctrl/avionics_cache/schemata'

# 3. 锁定为独占模式
sudo bash -c 'echo exclusive > /sys/fs/resctrl/avionics_cache/mode'
```


### 结果分析

从上述测试数据可以得出以下分析结果：

**1. 无负载基准表现：**
在无外部压力的隔离状态下，程序的系统抖动（Jitter）为 11 μs。这反映了在排除操作系统软件调度干扰后，处理器微架构固有的流水线波动情况，可作为性能对比的基准底噪。

**2. 共享缓存下的性能退化：**
在使用普通内存且未划分 L3 缓存分区的情况下，引入外部负载会导致最坏情况执行时间（WCET）大幅增加至 9941 μs，系统抖动达到 8959 μs。这是因为靶标程序的数据被外部进程频繁挤出 L3 缓存，导致缓存未命中（Cache Miss）率上升，程序被迫向主存请求数据，从而产生了显著的访存延迟。对于控制周期要求严格的实时系统而言，此类延迟会导致执行超时。

**3. 缓存隔离的效果验证：**
在同时启用 CMA 物理连续内存和 resctrl L3 缓存独占分区的状态下，面对相同的外部负载：
* **平均执行时间（Avg）：** 相比无负载状态仅增加 14 μs（从 456 μs 升至 470 μs），表明核心的计算性能基本未受干扰。
* **系统抖动（Jitter）：** 维持在 63 μs。相较于未保护状态下的 8959 μs，系统抖动降低了约 99.3%。这表明通过 resctrl 划分的专属 L3 缓存空间有效阻止了外部进程对缓存数据的擦写。

## 关于缓存分区的一些说明
针对在配置 `resctrl` 过程中可能产生的疑问，以下从硬件架构与内核机制角度进行补充说明。

###  为什么只隔离 L3 缓存，不管 L1 和 L2？

这与多核处理器的**物理层级结构 (Cache Hierarchy)** 密切相关。在现代 CPU（尤其是 AMD Ryzen 架构）中：

        每个物理核心都拥有专属的 L1 和 L2 缓存。例如，核心 10 的 L2 缓存与核心 0 的 L2 缓存从物理链路上就是隔绝的。即便核心 0 运行高强度的缓存填充任务，也无法访问或污染核心 10 的私有缓存，因此不存在“吵闹邻居”干扰。
        L3 缓存（Last Level Cache, LLC）是跨核心共享的。它是 CPU 内部的最后一道高速防线，当各核心的私有缓存装不下数据时，都会向 L3 申请空间。如果其他非隔离核心在进行海量访存（如 `stress-ng`），它们会迅速占满 L3 的所有条目，将航电程序的热点数据强制“挤”出缓存，导致性能剧烈抖动。

**结论：** 只有 L3 是共享争用资源，因此硬件隔离（Cache Partitioning）的重点只能也必须放在 L3。

###  为什么会有节点 0 和节点 1 (L3:0 和 L3:1)？

这源于 AMD 的 **CCX (Core Complex)** 微架构设计。

你的 Ryzen 5 5500U 虽然拥有 6 个物理核心，但其内部并非所有核心都直连在同一块 8MB L3 缓存上。其拓扑结构实际上是“拼图式”的：
* **CCX 0 (对应节点 L3:0)：** 包含 3 个物理核心，它们共同围绕并共享一块 4MB 的物理 L3 缓存。
* **CCX 1 (对应节点 L3:1)：** 包含另外 3 个物理核心，它们共享另一块独立的 4MB 物理 L3 缓存。

虽然 `cbm_mask` 显示为 `ffff`（即支持将缓存划分为 16 份），但这是针对**每一个物理缓存簇**而言的。Linux 内核识别到了两个独立的物理 L3 实例，因此出现了 `L3:0` 和 `L3:1`。你在节点 0 上修改掩码，只会影响 CCX 0 里的核心性能，而不会跨越到 CCX 1。

###  节点 1  的 0000 区域，后续还能分给其他分区吗？

**可以**

* **当前的静默状态：** 此时你在节点 1 上留下 `0000`（未使用的 4 个 bit）是因为当前的航电靶标（核心 10/11）恰好都属于 CCX 0。为了保证该节点内没有重叠冲突，我们让系统全局掩码退避了这 4 个 bit 的算力权重。


这种基于 `resctrl` 的方案，允许我们根据任务的物理拓扑位置，精细化地运营每一片缓存空间。

## 附录：代码整合模板

前面的代码分散在三个文件里，每次换方案都得重新编译另一个文件，比较麻烦。这里把它们合并成一个统一模板，通过文件头部几个 `#define` 开关来控制启用哪些分区功能。

### 设计思路

四个版本的核心差异就两个：调度策略和内存来源。把它们抽成条件编译：

| 开关 | 作用 | 对应原文件 |
| :--- | :--- | :--- |
| `USE_TIME_PARTITION` | 开启后应用 `SCHED_DEADLINE` 时间窗口 | `avionics_sche.c` |
| `USE_CMA_MEM` | 开启后从 `/dev/avionics_cma` 映射内存 | `avionics_mem.c` |

编译期开关比 `getopt` 运行时传参更适合这个场景——功能选定后编译器自动裁掉无用分支，零运行时开销，也更接近真实嵌入式系统的构建方式。

### `avionics_unified.c`

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>
#include <time.h>

// ==================== 功能开关 ====================
// 需要哪个功能就把哪行注释去掉（或编译时加 -D 定义）

// #define USE_TIME_PARTITION    // SCHED_DEADLINE 时间分区
// #define USE_CMA_MEM           // CMA 连续物理内存分区

// ==================== 时间分区相关 ====================
#ifdef USE_TIME_PARTITION
#include <unistd.h>
#include <sys/syscall.h>
#include <linux/sched.h>

#define SCHED_RUNTIME_MS  3
#define SCHED_DEADLINE_MS 10
#define SCHED_PERIOD_MS   10

struct sched_attr {
    uint32_t size;
    uint32_t sched_policy;
    uint64_t sched_flags;
    int32_t sched_nice;
    uint32_t sched_priority;
    uint64_t sched_runtime;
    uint64_t sched_deadline;
    uint64_t sched_period;
};

int sched_setattr(pid_t pid, const struct sched_attr *attr, unsigned int flags) {
    return syscall(SYS_sched_setattr, pid, attr, flags);
}

static void setup_time_partition(void) {
    struct sched_attr attr = {0};
    attr.size = sizeof(attr);
    attr.sched_policy = 6;
    attr.sched_runtime  = SCHED_RUNTIME_MS * 1000000ULL;
    attr.sched_deadline = SCHED_DEADLINE_MS * 1000000ULL;
    attr.sched_period   = SCHED_PERIOD_MS * 1000000ULL;

    if (sched_setattr(0, &attr, 0) == -1) {
        perror("SCHED_DEADLINE 设置失败 (需 root 并在 cgroups 分区内运行)");
        exit(EXIT_FAILURE);
    }
    printf("=> 时间分区: 每 %dms 周期运行 %dms\n", SCHED_PERIOD_MS, SCHED_RUNTIME_MS);
}
#endif

// ==================== 内存分区相关 ====================
#ifdef USE_CMA_MEM
#include <fcntl.h>
#include <string.h>
#include <sys/mman.h>

#define ALLOC_SIZE  (2 * 1024 * 1024)
#define TOTAL_SIZE  (512 * 1024 * 1024)

static void *cma_base = NULL;

static void* setup_cma_memory(void) {
    if (mlockall(MCL_CURRENT | MCL_FUTURE) == -1) {
        perror("mlockall 失败");
    }

    int fd = open("/dev/avionics_cma", O_RDWR);
    if (fd < 0) {
        perror("无法打开 /dev/avionics_cma");
        exit(EXIT_FAILURE);
    }

    void *ptr = mmap(NULL, TOTAL_SIZE, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    close(fd);

    if (ptr == MAP_FAILED) {
        perror("CMA mmap 失败");
        exit(EXIT_FAILURE);
    }

    memset(ptr, 0, 3 * ALLOC_SIZE);
    printf("=> 内存分区: 已映射 512MB CMA 连续物理内存\n");
    return ptr;
}

static void teardown_cma_memory(void) {
    if (cma_base) munmap(cma_base, TOTAL_SIZE);
}
#endif

// ==================== 核心计算逻辑 ====================
#define NUM_ITERATIONS 10000
#define MATRIX_SIZE 50

static void matrix_multiply(float A[MATRIX_SIZE][MATRIX_SIZE],
                            float B[MATRIX_SIZE][MATRIX_SIZE],
                            float C[MATRIX_SIZE][MATRIX_SIZE]) {
    for (int i = 0; i < MATRIX_SIZE; i++) {
        for (int j = 0; j < MATRIX_SIZE; j++) {
            C[i][j] = 0.0;
            for (int k = 0; k < MATRIX_SIZE; k++) {
                C[i][j] += A[i][k] * B[k][j];
            }
        }
    }
}

int main() {
#ifdef USE_TIME_PARTITION
    setup_time_partition();
#endif

    // 分配矩阵内存
    float (*A)[MATRIX_SIZE];
    float (*B)[MATRIX_SIZE];
    float (*C)[MATRIX_SIZE];

#ifdef USE_CMA_MEM
    cma_base = setup_cma_memory();
    size_t matrix_bytes = MATRIX_SIZE * MATRIX_SIZE * sizeof(float);
    char *mem = (char *)cma_base;
    A = (float (*)[MATRIX_SIZE])(mem);
    B = (float (*)[MATRIX_SIZE])(mem + matrix_bytes);
    C = (float (*)[MATRIX_SIZE])(mem + 2 * matrix_bytes);
#else
    A = malloc(MATRIX_SIZE * MATRIX_SIZE * sizeof(float));
    B = malloc(MATRIX_SIZE * MATRIX_SIZE * sizeof(float));
    C = malloc(MATRIX_SIZE * MATRIX_SIZE * sizeof(float));
    if (!A || !B || !C) {
        perror("内存分配失败");
        exit(EXIT_FAILURE);
    }
#endif

    // 填充随机数据
    for (int i = 0; i < MATRIX_SIZE; i++) {
        for (int j = 0; j < MATRIX_SIZE; j++) {
            A[i][j] = (float)(rand() % 100) / 10.0;
            B[i][j] = (float)(rand() % 100) / 10.0;
        }
    }

    printf("=> 矩阵大小: %dx%d, 循环次数: %d\n", MATRIX_SIZE, MATRIX_SIZE, NUM_ITERATIONS);

    // 预热
    for (int i = 0; i < 10; i++)
        matrix_multiply(A, B, C);

    // 正式测试
    struct timespec start, end;
    uint64_t elapsed_us, max_time_us = 0, total_time_us = 0;

    for (int iter = 0; iter < NUM_ITERATIONS; iter++) {
        clock_gettime(CLOCK_MONOTONIC, &start);
        matrix_multiply(A, B, C);
        clock_gettime(CLOCK_MONOTONIC, &end);

        elapsed_us = (end.tv_sec - start.tv_sec) * 1000000 +
                     (end.tv_nsec - start.tv_nsec) / 1000;
        if (elapsed_us > max_time_us) max_time_us = elapsed_us;
        total_time_us += elapsed_us;
    }

    uint64_t avg_time_us = total_time_us / NUM_ITERATIONS;
    printf("\n测试完成！\n");
    printf("------------------------------------------------\n");
    printf("平均执行时间 (Average): %llu 微秒 (us)\n", (unsigned long long)avg_time_us);
    printf("最坏情况执行时间 (WCET):  %llu 微秒 (us)\n", (unsigned long long)max_time_us);
    printf("系统抖动 (Jitter):      %llu 微秒 (us)\n", (unsigned long long)(max_time_us - avg_time_us));
    printf("------------------------------------------------\n");
    printf("[防优化输出] C[0][0] = %f\n", C[0][0]);

#ifdef USE_CMA_MEM
    teardown_cma_memory();
#else
    free(A); free(B); free(C);
#endif

    return 0;
}
```

### 编译与使用

```bash
gcc -O0 avionics_unified.c -o avionics_unified
```

上面只开了一个 CMA 开关的话，等价于原来的 `avionics_mem`。要启用其他功能，把对应 `#define` 注释去掉即可，也可以通过 `-D` 在命令行指定：

```bash
# 开启时间分区
gcc -O0 -DUSE_TIME_PARTITION avionics_unified.c -o avionics_unified

# 时间分区 + 内存分区全开
gcc -O0 -DUSE_TIME_PARTITION -DUSE_CMA_MEM avionics_unified.c -o avionics_unified
```

> **注意：** 时间分区和 CMA 内存分区都需要配套的系统环境——时间分区要求终端已在 cgroups 分区内，CMA 要求 `avionics_cma.ko` 已加载。CPU 隔离和 L3 缓存分区属于系统级配置，只需把进程放进对应的 cgroup/resctrl 资源组即可，代码无需感知。
