# 可视化验证方法

CPU 调度发生在微秒级别，终端里打印的 Avg/WCET/Jitter 只能看到结果，看不到过程。要验证核心隔离和时间窗口是否真的在硬件层面生效，需要用 `trace-cmd` 录制内核调度事件，再用 KernelShark 图形化回放。

---

## 安装工具

```bash
sudo apt update
sudo apt install trace-cmd kernelshark
```

---

## 录制调度事件

新开一个终端（避免录制工具自身的调度事件污染数据），执行：

```bash
sudo trace-cmd record \
    -e sched:sched_switch \
    -e sched:sched_wakeup \
    -e sched:sched_wakeup_new
```

三个事件的作用：

| 事件 | 作用 |
| :--- | :--- |
| `sched_switch` | 进程上下台的时间点，两次 switch 之间的差值就是实际占据 CPU 的时间 |
| `sched_wakeup` | 进程从休眠被唤醒、放入就绪队列的瞬间，可计算调度延迟 |
| `sched_wakeup_new` | 新进程首次被唤醒，不记录的话会错过程序启动瞬间 |

录制期间，在另一个终端运行待测程序。程序跑完后回到录制终端按 `Ctrl+C`，生成 `trace.dat`。

---

## 用 KernelShark 查看

```bash
kernelshark trace.dat
```

图形界面里，每个 CPU 核心对应一行横条。色块=进程正在运行，空白=idle。在 Search 框输入进程名可以精准定位。

## 关键分析方法

### 验证核心隔离

在非隔离核心（如 CPU 0-9）的行上，可以看到密集的色块和黑线（频繁的上下文切换）。隔离核心（如 CPU 10-11）的行几乎为空，只有目标进程的零星色块。对比即可确认隔离生效。

### 验证 SCHED_DEADLINE 时间窗口

在 KernelShark 中搜索进程名，选取相邻的几个 `sched_switch` 事件，读时间戳计算：

以设置 Runtime=3ms、Period=10ms 为例，预期结果：

- **实际运行时间：** 切换上台 → 切换下台的时间差 ≈ 3ms
- **强制休眠时间：** 下台 → 下次上台的时间差 ≈ 7ms
- **总周期：** 3ms + 7ms = 10ms

如果时间戳差值跟设置参数基本吻合（误差在微秒级），说明 SCHED_DEADLINE 的时间窗口在硬件层面确实生效了。

---

## 示例：提取一个完整周期

| 事件类型 | CPU | 时间戳 (s) | 详情 |
| :--- | :--- | :--- | :--- |
| `sched_switch` | 10 | 1627.5632 | idle → avionics_sche |
| `sched_switch` | 10 | 1627.5663 | avionics_sche → idle |
| `sched_switch` | 11 | 1627.5732 | idle → avionics_sche |

- Runtime: 1627.5663 - 1627.5632 = 0.0031s = 3.1ms
- Sleep: 1627.5732 - 1627.5663 = 0.0069s = 6.9ms
- Period: 10.0ms

与配置的 3ms/10ms 一致。
