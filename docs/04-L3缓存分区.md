# L3 缓存分区

CPU 核心和物理内存都隔离了，还剩一个更隐蔽的干扰：L3 缓存踩踏。核心 10/11 的热数据刚加载进 L3，就被核心 0-9 上 stress-ng 的垃圾数据顶掉，Cache Miss 激增，WCET 拉出大毛刺。

通过 Linux 的 `resctrl` 接口调用 AMD/Intel 的 CAT（Cache Allocation Technology）硬件功能，可以直接在硅片级别为核心 10/11 划出专属 L3 缓存块。

> cgroups 管软件调度，resctrl 管硬件微架构，两者互不冲突。合在一起才是完整的硬实时分区。

---

## 1. 挂载 resctrl 并确认硬件能力

```bash
sudo mount -t resctrl resctrl /sys/fs/resctrl
cd /sys/fs/resctrl
ls info/
```

<img width="682" height="88" alt="image" src="https://github.com/user-attachments/assets/099cffa2-8f8a-4226-9664-d0041dfc19f4" />

`info/` 下有四个条目：

| 条目 | 用途 |
| :--- | :--- |
| **L3** | L3 缓存分配（CAT），缓存分区的核心接口 |
| **L3_MON** | L3 缓存监控（RDT），可实时查看进程缓存占用 |
| **MB** | 内存带宽分配（MBA），限制非关键任务的访存带宽 |
| **last_cmd_status** | 排错文件，每次写入操作后显示 ok 或具体失败原因 |

查看缓存容量位掩码：

```bash
cat info/L3/cbm_mask
# 结果: ffff
```

`ffff` = 16 bit。5500U 共 8MB L3，每个 bit 控制 8MB / 16 = 512KB。计划把最右侧 4 bit（`f`，共 2MB）划为航电专属。

---

## 2. 修改全局掩码

默认根目录的 `schemata` 对 L3 是全量开放的（`ffff`），如果直接创建新组并分配，内核会报 `Schemata overlaps`，拒绝开启独占模式。

另外 Ryzen 5500U 有两个 CCX，对应两个 L3 实例（L3:0 和 L3:1），两边都得退让：

```bash
sudo bash -c 'echo "L3:0=fff0;1=fff0" > /sys/fs/resctrl/schemata'
```

这一步把系统全局进程限制在前 12 bit（`fff0`），把尾部 4 bit 腾空。

> **踩坑记录：** 一开始只改了 L3:0=fff0，漏了 L3:1 还是 ffff。写入 `exclusive` 时报 `无效的参数`，查 `cat info/last_cmd_status` 显示 `Schemata overlaps`，定位到节点 1 还占着那 4 bit。两边同步改成 fff0 后正常。

## 3. 创建航电资源组

```bash
# 1. 创建资源组
sudo mkdir /sys/fs/resctrl/avionics_cache

# 2. 分配专属掩码（分区挂在 L3:0 上，L3:1 不做分配）
sudo bash -c 'echo "L3:0=f;1=0" > /sys/fs/resctrl/avionics_cache/schemata'

# 3. 锁定为独占模式
sudo bash -c 'echo exclusive > /sys/fs/resctrl/avionics_cache/mode'
```

---

## 4. 绑定核心

```bash
sudo bash -c 'echo "10-11" > /sys/fs/resctrl/avionics_cache/cpus_list'
```

从此刻起，核心 10 和 11 发出的所有访存请求只能命中那 2MB 专属 L3 区域。外部核心的缓存踩踏碰不到这块空间。

## 5. 效果验证

### 5.1 静态验证

```bash
cat /sys/fs/resctrl/info/L3/bit_usage
# 结果: 0=SSSSSSSSSSSSEEEE;1=SSSSSSSSSSSS0000
```

- **L3:0：** 前 12 个 `S`（Shareable）系统公共，后 4 个 `E`（Exclusive）航电独占
- **L3:1：** 后 4 个 `0`（Unused）是全局掩码腾出的隔离带

```bash
cat /sys/fs/resctrl/avionics_cache/cpus_list
# 结果: 10-11
```

## 6. 补充说明

### 为什么只隔离 L3，不管 L1 和 L2？

L1 和 L2 是每个物理核心私有的。核心 10 的 L1/L2 与核心 0 的 L1/L2 在物理链路上就是隔绝的，不存在跨核心污染。只有 L3（LLC）是跨核心共享的，是真正的争用资源，因此缓存分区的重点只能是 L3。

### 为什么有 L3:0 和 L3:1 两个节点？

Ryzen 5 5500U 内部有两个 CCX（Core Complex），每个 CCX 包含 3 个物理核心和一块独立的 4MB L3：
- **CCX 0（L3:0）：** 核心 0/2/4/6/8/10（逻辑核），共享 4MB L3
- **CCX 1（L3:1）：** 核心 1/3/5/7/9/11（逻辑核），共享 4MB L3

`cbm_mask` 显示的 `ffff` 是每个物理缓存簇各自的划分粒度，Linux 内核识别到了两个独立 L3 实例。在 L3:0 上修改掩码只影响 CCX 0 里的核心。

### L3:1 那 4 个 bit 后续还能用吗？

可以。当前航电靶标的核心 10/11 都在 CCX 0 内，所以 L3:1 上留了 `0000`。如果将来有其他分区部署到 CCX 1 的核心上，那 4 个 bit 可以分给它。
